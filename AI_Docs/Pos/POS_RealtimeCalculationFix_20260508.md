# POS Realtime Calculation Fix - 2026-05-08

## Scope

إصلاح regression في شاشة:

- `/Pos/PosTransaction/Index`

التعديل يخص UX والتحقق والحسابات الفورية فقط. لم يتم تعديل SQL أو إجراءات الحفظ أو القيود أو الترحيل.

## Root Cause

تم رصد سببين:

- تمييز الحقول المطلوبة كان يعتبر `.pos-step-locked` حالة إخفاء، فحقل مثل رقم المحفظة قد يتم تجاهله أو لا يحصل على focus/scroll رغم ظهور رسالة الخطأ.
- حساب العمولة/المبلغ المطلوب سحبه من الماكينة كان يمر عبر debounce افتراضي 400ms، لذلك كان يظهر للمستخدم كرقم متأخر أو stale أثناء الكتابة.

## Event Pipeline

تم إضافة مسار مركزي:

- `recalculateInvoiceSummary(options)`

هذا المسار يقوم فوريا بـ:

- إعادة حساب صفوف الأصناف.
- تحديث الإجمالي، الرسوم، المدفوع، والمتبقي.
- تحديث بطاقة ملخص كاش أوت والمبلغ المطلوب سحبه من الماكينة.
- طلب حساب العمولة بدون تأخير عند تغيير الحقول المؤثرة.

## Affected Fields And Events

يتم تشغيل الحساب الفوري عند:

- تغيير `rechargeValue`.
- تغيير `violationValue`.
- تغيير `payedValue`.
- تعديل كمية/سعر/ضريبة أي صف.
- حذف صف خدمة.
- تغيير نوع الخدمة الأساسي.
- تغيير المحفظة/البنك الفرعي.
- تحميل صنف الخدمة الافتراضي.

## Machine Withdrawal Flow

في كاش أوت:

- عند تغيير المبلغ يتم تصفير قيمة السحب القديمة حتى لا يظهر رقم stale.
- يتم عرض تقدير فوري = قيمة الشحن + آخر عمولة ماكينة معروفة.
- عند وصول نتيجة العمولة من السيرفر يتم استبدال التقدير بالقيمة النهائية `CashOutMachineWithdrawalAmount`.
- لا يحتاج المستخدم إلى refresh أو blur حتى يرى تغير الرقم.

## Validation Fix

تم تحسين ربط حقل رقم المحفظة:

- `tetNumPoket`
- `violationWalletNo`

وتم إضافة `data-pos-field="WalletNumber"` لهما في view.

تم تعديل visibility helper بحيث لا يعتبر `pos-step-locked` حقل مخفي. هذا يسمح بالآتي:

- scroll إلى الحقل.
- focus عليه.
- إضافة red border/glow.
- إضافة رسالة inline.
- إزالة التمييز فور إدخال قيمة صحيحة.

## Debugging

تمت إضافة logging آمن لا يعمل إلا عند:

- فتح الشاشة مع `?posDebug=1`
- أو تفعيل `localStorage.POS_DEBUG = "1"`

المفاتيح:

## Regression Prevention Notes

- أي تغيير في قيمة مؤثرة على الإجماليات يجب أن يمر من `recalculateInvoiceSummary`.
- لا يتم الاعتماد على debounce لحسابات الشاشة الأساسية.
- طلبات العمولة async مسموحة، لكن واجهة المستخدم تعرض قيمة فورية وتستبدلها بالقيمة النهائية عند وصول الرد.
- لا يتم استخدام أرقام ماكينة قديمة بعد تغيير مبلغ كاش أوت.

## Validation Checklist

- ترك رقم المحفظة فارغا في كاش أوت يميز الحقل ويركز عليه.
- كتابة رقم المحفظة تزيل التمييز مباشرة.
- إدخال مبلغ الشحن يحدث الإجمالي ومبلغ السحب فوريا.
- تعديل الرسوم أو الضريبة يحدث الإجماليات فوريا.
- حذف صف خدمة يحدث الإجماليات فوريا.
- تغيير الخدمة أو البنك يطلب العمولة مباشرة.
- الحفظ الناجح لا يزال يستخدم نفس flow القديم بدون تغيير محاسبي.
