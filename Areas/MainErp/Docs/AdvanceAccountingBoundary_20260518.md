## Confirmed Flow

1. `FrmEmpsAdvanceRequest.frm` saves and sends the advance request for approval through `SendTopost`.
2. `FrmEmpsAdvance.frm` creates the actual advance in `TblEmpAdvance` and installment rows in `TblEmpAdvanceDetails`.
3. `TblEmpAdvance.orderNO` is the confirmed link back to `TblEmpAdvanceRequest.AdvanceID`.
4. The direct `Notes` / `DOUBLE_ENTREY_VOUCHERS` creation block for the actual advance is present in VB6 as commented code, not an executable confirmed posting path.
5. `FrmEmpsAdvancePayed1.frm` changes installment status fields such as `Payed`, `StutsID`, and `EmpAdPaID`; its direct accounting block is also commented.
6. The confirmed accounting impact for installments is through payroll posting when `TotalAdvance` is included in the salary run.
7. `FrmEmpsAdvance.frm` can create opening-balance entries in `DOUBLE_ENTREY_VOUCHERS1` when opening balance options are used. This is not a cash payment voucher.

## Web Boundary Implemented

The shared HR service now exposes `GetAdvanceAccountingBoundary(requestId)`.

Routes:
- `/Pos/Hr/AdvanceAccountingBoundary?id=...`
- `/MainErp/Hr/AdvanceAccountingBoundary?id=...`

The boundary reports:
- actual advance link
- direct journal trace count
- opening journal trace count
- payroll deduction link count
- posted payroll deduction link count
- whether the trace is unsupported or needs review

## Posting Rule

The Advances screen does not create a guessed accounting entry and does not create a guessed cash/payment voucher.

Confirmed states:
- Request approval: HR workflow only.
- Actual advance creation: `TblEmpAdvance` + `TblEmpAdvanceDetails` only.
- Payroll deduction: posted through payroll journal.
- Opening balance: only if historical data has `opening_balance_voucher_id`.

## Unsupported Edge Cases

- Creating a new `سند صرف` directly from advance disbursement is not enabled because the VB6 execution path for that linkage was not confirmed.
- Reversing a direct historical advance journal is not implemented in HR. If a direct trace exists in `DOUBLE_ENTREY_VOUCHERS`, it must be reviewed from the accounting/journal screen first.
- Opening-balance entries are reported but not regenerated by the web advance workflow.

## Double Posting Protection

- Disbursement checks `TblEmpAdvance.orderNO` under transaction lock before inserting the actual advance.
- Existing actual advance blocks a second disbursement.
- Payroll deduction links use `PayrollRunAdvanceDeductions` and mark posted installments from payroll posting, not from HR advance disbursement.
