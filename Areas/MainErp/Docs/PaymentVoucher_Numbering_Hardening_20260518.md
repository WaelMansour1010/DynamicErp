## Implementation

Changed script:

- `F:\Source Code\DynamicErp\Areas\MainErp\Sql\16_PaymentVoucher_VB6_SaveParity.sql`

Objects changed/added:

| Object | Change |
| --- | --- |
| `dbo.SerialCounters_V2` | Existing counter table used as the only runtime serial allocator. |
| `dbo.usp_DynamicErpVoucher_SeedSerialCounters` | New seed procedure. Snapshots existing `Notes` tails into counters during deployment. |
| `dbo.usp_DynamicErpVoucher_NextSerial` | Rebuilt to allocate only from `SerialCounters_V2` under transaction lock. No runtime `MAX(...)+1` from `Notes`. |
| `dbo.usp_DynamicErpVoucher_Save` | Continues to call `dbo.usp_DynamicErpVoucher_NextSerial` only on create. Edit keeps existing serials. Delete does not touch counters. |

Concurrency controls:

- Requires an open transaction; returns error if called outside a transaction.
- Uses `sys.sp_getapplock` with `LockOwner='Transaction'` per serial scope.
- Updates `SerialCounters_V2` with `UPDLOCK, HOLDLOCK`.
- If a counter is missing but existing data exists in that scope, generation fails instead of guessing from `MAX`.
- For a genuinely new scope/new period with no existing data, creates the counter at `start_at - 1`, then increments inside the same transaction.
- Rollback returns the counter to the prior value because allocation is done in the caller transaction.

Intentional modernization:

- VB6 scans existing rows with `MAX(...)` to derive the next number. The web path now uses a counter row instead.
- Existing data is read only during deployment seeding, not during runtime save.
- `numbering_id=1` is treated as genuinely continuous instead of resetting by year.

## Remaining Scope

This pass fixes `NoteSerial` / `NoteSerial1` voucher numbering. It does not replace legacy `MAX(...)+1` allocators for unrelated numeric IDs such as `Notes.NoteID`, `DOUBLE_ENTREY_VOUCHERS.Double_Entry_Vouchers_ID`, or `DEV_ID_Line_No1`; those should be handled in a separate ID allocator hardening task.
