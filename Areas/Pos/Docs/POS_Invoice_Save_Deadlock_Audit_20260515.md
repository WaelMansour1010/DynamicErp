# POS Invoice Save Deadlock Audit - 2026-05-15

## Scope

Area: `Areas/Pos`

Production issue: POS/Kishny invoice saves under peak usage show deadlocks/contention more frequently than the legacy VB6 flow.

This audit uses the 2026-05-14 investigation as baseline. It does not propose copying VB6 blindly. The useful lesson is that VB6 was looser and less centralized, which accidentally avoided some contention. The target is to preserve accounting correctness while reducing unnecessary serialization and lock duration.

No code, SQL procedure, constraint, or production behavior was changed by this audit.

- `DEV_Serial` is not a durable business key. It is nullable, duplicated, not unique, and not indexed as a key.

## Executive Conclusion

The strongest current suspect is not `DOUBLE_ENTREY_VOUCHERS` row insertion by itself.

The likely contention pattern is:

1. .NET enters one large SQL transaction.
2. It allocates `Transaction_ID`, invoice `NoteSerial1`, `NoteID`, `NoteSerial`, `Double_Entry_Vouchers_ID`, and issue voucher IDs/serials inside that transaction.
3. It performs many account/settings/history lookups inside the same transaction.
4. It inserts invoice rows, notes, accounting rows, and issue voucher rows before commit.
5. Voucher coding uses a stricter centralized counter path than VB6, especially for invoice `Transaction_Type = 21`, `Sanad_No = 7`.

VB6 did not prove better correctness. It simply used looser `MAX + 1`/recordset behavior and fewer centralized locks. That reduced serialization pressure but allowed gaps and historical duplicates.

## VB6 Save Flow Summary

Primary files inspected:

- `F:\Source Code\SatriahMain\Cayshny\Frm\FrmSaleBill6.frm`
- `F:\Source Code\SatriahMain\Cayshny\Bas\ModAccounts.bas`
- `F:\Source Code\SatriahMain\Cayshny\Bas\Mod_DataBaseFunctions.bas`
- `F:\Source Code\SatriahMain\Cayshny\Bas\registry04082020.bas`

Important VB6 flow:

1. `SaveData` starts `Cn.BeginTrans`.
2. New invoices allocate `Transactions.Transaction_ID` using `new_id("Transactions", "Transaction_ID", "", True)`.
3. Invoice number is generated by `Voucher_coding(... Sanad_No=7, NoteType=170, Transaction_Type=21 ...)`.
4. Header is saved through ADODB recordset update.
5. Existing details/payments are deleted for edits, then details/payments are written.
6. `CreateNotes` writes the accounting note for the invoice.
7. `PG` creates invoice accounting rows.
8. `PG` allocates `LngDevID = new_id("DOUBLE_ENTREY_VOUCHERS", "Double_Entry_Vouchers_ID", "")`.
9. `PG` calls `ModAccounts.AddNewDev(...)` repeatedly, one journal detail line at a time.
10. `CreateIssueVoucher` / `CreateIssueVoucher2` creates stock issue transaction/type 19 logic separately in the VB6 flow.
11. `Cn.CommitTrans` commits the save.

VB6 serial behavior:

- `new_id` now attempts `dbo.GetNextID_FromSequence` on SQL Server, with a fallback to `SELECT MAX(...) + 1 WITH (NOLOCK)`.
- `Voucher_coding` reads `sanad_numbering` and then uses `MAX(NoteSerial1)` style queries filtered by branch/type/date/month/prefix/store rules.
- No explicit `UPDLOCK`, `HOLDLOCK`, `SERIALIZABLE`, `TABLOCKX`, or `sp_getapplock` was found in the inspected VB6 `Voucher_coding` path.
- `AddNewDev` accepts optional `StrDEV_Serial`; if blank, it calls `GetNewDEV_Serial`.
- `GetNewDEV_Serial` counts distinct `Double_Entry_Vouchers_ID` for the same `RecordDate`, then formats a date-prefixed display value.

VB6 did not lock:

- It did not globally serialize `NoteSerial1` with a counter row.
- It did not protect `DEV_Serial` as a strict sequence.
- It did not use an explicit table lock for `DOUBLE_ENTREY_VOUCHERS` insertion in the inspected `AddNewDev` path.

## .NET Save Flow Summary

Current web path:

1. UI: `Areas/Pos/Scripts/pos-transaction.js`
2. Controller: `Areas/Pos/Controllers/PosTransactionController.cs`
3. Repository: `Areas/Pos/Data/PosSqlRepository.cs`
4. Stored procedure: `Areas/Pos/Sql/30_POS_SaveTransaction_UnicodeText.sql`
5. SQL object: `dbo.usp_POS_SaveTransaction`

Important .NET flow:

1. Repository calls `dbo.usp_POS_SaveTransaction`.
2. Procedure begins one SQL transaction near line 228.
3. Edit mode locks existing transaction via `Transactions WITH (UPDLOCK, HOLDLOCK)`.
4. It deletes old issue/accounting/detail/payment rows on edit.
5. It allocates invoice `Transaction_ID` through `dbo.GetNextID_FromSequence`.
6. It allocates invoice `NoteSerial1` through `dbo.usp_Voucher_coding_V2`.
7. It validates token/card duplicate conditions and may take a card-token app lock.
8. It inserts `Transactions`, `Transaction_Details`, and `TblSalesPayment`.
9. It allocates invoice accounting `NoteID` through `GetNextID_FromSequence`.
10. It allocates invoice accounting `NoteSerial` through `dbo.usp_Notes_coding_V2`.
11. It allocates `Double_Entry_Vouchers_ID` through `GetNextID_FromSequence`.
12. It performs account/settings/history lookups for accounting line construction.
13. It inserts `Notes`.
14. It updates `Transactions.NoteId` and `Transactions.NoteSerial`.
15. It inserts `DOUBLE_ENTREY_VOUCHERS`.
16. It loops issue-voucher stores and creates type 19 `Transactions`, `Notes`, and `Transaction_Details`.
17. It commits, then writes stage logs outside the main transaction.

## Lock Order Findings

Current .NET effective lock/order:

| Order | Operation | Main objects |
| ---: | --- | --- |
| 1 | Begin transaction | transaction scope opens |
| 2 | Edit lock/delete old rows | `Transactions`, `DOUBLE_ENTREY_VOUCHERS`, `Notes`, `Transaction_Details`, `TblSalesPayment` |
| 3 | Invoice ID allocation | `GetNextID_FromSequence`, sequence/app lock |
| 4 | Invoice voucher coding | `usp_Voucher_coding_V2`, `SerialCounters_V2` |
| 5 | Card/token validation/app lock | `TblCusCsh`, `Transactions`, `Transaction_Details`, app lock |
| 6 | Header/details/payments insert | `Transactions`, `Transaction_Details`, `TblSalesPayment` |
| 7 | Accounting note/id/serial allocation | `Notes`, `usp_Notes_coding_V2` |
| 8 | Journal ID allocation | `GetNextID_FromSequence` for `DOUBLE_ENTREY_VOUCHERS` |
| 9 | Accounting lookups | branches/accounts/users/options/items/history |
| 10 | Accounting insert | `Notes`, `DOUBLE_ENTREY_VOUCHERS` |
| 11 | Issue voucher loop | `Transactions`, `Notes`, `Transaction_Details`, voucher coding |
| 12 | Final update and commit | `Transactions` |

VB6 approximate order:

| Order | Operation | Main objects |
| ---: | --- | --- |
| 1 | Begin transaction | connection transaction |
| 2 | Header recordset and invoice serial | `Transactions`, `Voucher_coding` reads |
| 3 | Details/payments | `Transaction_Details`, `TblSalesPayment` |
| 4 | `CreateNotes` | `Notes` |
| 5 | `PG` accounting | account lookup reads, `DOUBLE_ENTREY_VOUCHERS` rows |
| 6 | Issue voucher path | type 19 transaction/details/notes |
| 7 | Commit | connection transaction |

Main difference:

The .NET procedure centralizes all allocation and validation inside one stored procedure transaction and uses stricter allocation locks. VB6 had a broad transaction too, but its serial paths were mostly read-based and did not impose the same counter-row/app-lock discipline.

## Reads And Lookups Inside The .NET Transaction

The following are currently inside the `usp_POS_SaveTransaction` transaction and are candidates for review:

| Category | Examples | Risk |
| --- | --- | --- |
| Branch/account lookup | `TblBranchesData`, `ACCOUNTS`, `get_account_code_branch`, `GetMyAccountCode` | Extends transaction while doing reference-data reads. |
| Historical account inference | `DOUBLE_ENTREY_VOUCHERS` joined to `Transactions` to infer customer/wallet accounts | Reads large journal table inside write transaction; can overlap with ledger/report reads and DEV writes. |
| User cash/bank lookup | `TblUsers`, `TblBoxesData`, `BanksData` | Reference data read inside lock window. |
| Option/settings lookup | `TblOptions`, `TblSettsReqLimK` | Stable settings should usually be resolved before critical writes or cached. |
| Item/customer defaults | `TblItems`, `TblCustemers` | Reference lookup inside write transaction; may be movable if inputs are validated. |
| Price/cost function | `CheckPriceRangeSales3(...)` | Function call inside transaction can hide reads and increase duration. |
| Issue-voucher cost read | Reads just-inserted `Transaction_Details` to compute issue cost | Must stay consistent but can possibly use table variables or already-known request data. |
| Issue store cursor | Cursor over saved details | Keeps transaction open while allocating/writing issue voucher rows. |

Safe principle:

Reference lookups that do not depend on rows created during the transaction should be resolved before `BEGIN TRANSACTION`, cached, or moved into a short preflight step. Writes and accounting inserts must remain atomic unless a formal redesign is approved.

## DOUBLE_ENTREY_VOUCHERS ID/Serial Hotspot Analysis

### VB6 Behavior

- `PG` allocates `LngDevID` via `new_id("DOUBLE_ENTREY_VOUCHERS", "Double_Entry_Vouchers_ID", "")`.
- `AddNewDev` writes individual journal rows using the caller-provided `LngDevID` and line number.
- `DEV_Serial` is optional. If omitted, `AddNewDev` calls `GetNewDEV_Serial`.
- `GetNewDEV_Serial` reads distinct voucher IDs for a date and formats a display value.
- No unique constraint/index behavior was found that makes `DEV_Serial` a durable identity.

### .NET Behavior

- `dbo.usp_POS_SaveTransaction` allocates `Double_Entry_Vouchers_ID` via `dbo.GetNextID_FromSequence`.
- It inserts journal lines in one set-based `INSERT INTO dbo.DOUBLE_ENTREY_VOUCHERS`.
- Yesterday’s `97_POS_DEVSerial_RemoveFromHotPath.sql` removes `POS_DEVSerialAllocator` from the POS save path and assigns display text from date + voucher id.

### Current ID Allocation Method

- `GetNextID_FromSequence` uses `sys.sp_getapplock` with an exclusive session lock by table/field resource.
- It uses SQL Server sequence objects, `NO CACHE`, and fallback/repair checks with `MAX(...) WITH (HOLDLOCK, UPDLOCK)`.
- Allocation is global by table/field, not branch/store/date/type scoped.

### Lock Risk

`DOUBLE_ENTREY_VOUCHERS` has two risks:

1. Allocation and insert are not currently the measured dominant slow stages.
2. The table is still high-risk because it is huge, heavily indexed, and touched by reports/accounting reads.

Therefore it should not be cleared as harmless, but it should not be the first production fix target unless real deadlock XML names a DEV index/page/key.

### Index Risk

Prior diagnostics found:

- Clustered PK on `(Double_Entry_Vouchers_ID, DEV_ID_Line_No)`.
- Many nonclustered indexes, some wide.
- Page latch/write amplification evidence on the PK and selected NC indexes.
- No triggers on `DOUBLE_ENTREY_VOUCHERS`.
- FKs to `ACCOUNTS`, `Notes`, and `ReceiptQest`, with cascade behavior on some relationships.

Risk: insert maintenance touches multiple index structures; report/ledger reads can still participate in deadlocks if predicates or isolation conflict.

### Deadlock Scenarios

Most plausible DEV-related scenarios:

- POS save holds `Transactions`/`Notes` locks, then inserts `DOUBLE_ENTREY_VOUCHERS`; a report or accounting screen reads DEV first and then joins to `Notes`/`Transactions`.
- Manual journal/payment paths allocate `MAX(Double_Entry_Vouchers_ID)+1 WITH (UPDLOCK,HOLDLOCK)` while POS save uses `GetNextID_FromSequence`, creating different allocation/lock patterns.
- Cascading parent locks on `Notes` or `ACCOUNTS` overlap with journal insert validation.
- Page/index latch pressure becomes visible under production IO and concurrent ledger/report load, even though local isolated insert tests were fast.

### Safest Phase 1 Mitigation

- Keep accounting posting synchronous and atomic.
- Keep `Double_Entry_Vouchers_ID` and line numbers unchanged.
- Preserve the DEV insert logic.
- Use production deadlock XML and `POS_SaveAllocationStageLog` to prove whether DEV indexes are in the wait cycle.
- Prefer fixing voucher coding and transaction duration first.

### Deeper Phase 2 Redesign

- Standardize all POS-adjacent journal writers on `GetNextID_FromSequence` or another approved allocator. Avoid mixed `MAX+1 WITH (UPDLOCK,HOLDLOCK)` paths.
- Review DEV/report indexes only from actual production query/deadlock evidence.
- Consider report isolation/indexing changes if reports are blocking writes.
- Do not move accounting async unless business accepts delayed journal visibility and a reconciliation/retry model.

## Voucher Coding / NoteSerial Findings

`NoteSerial1` is not disposable:

- It appears in UI, invoice search, reports, printing, and descriptions.
- It should remain readable and should avoid new duplicates inside the approved business scope.

But it is not proven to require:

- global uniqueness,
- company-wide serialization,
- gap-free numbering,
- store scope when `sanad_numbering.StoreCoding = 0`.

Current bottleneck:

- `dbo.usp_POS_SaveTransaction` calls `dbo.usp_Voucher_coding_V2` for invoice `Transaction_Type = 21`, `Sanad_No = 7`.
- `usp_Voucher_coding_V2` calls `usp_GetNextSerial_V2`.
- The counter path updates `SerialCounters_V2` with `UPDLOCK, SERIALIZABLE`.
- Full-save stage logs showed invoice voucher coding as the p99 hotspot.

Safe direction:

- Preserve the visible format.
- Use the real legacy scope: branch/type/month/prefix and store only when `StoreCoding = 1`.
- Deploy/enable branch-scoped setting only after production preflight.
- Keep default `Company` until a reviewed rollout changes `TblOptions.POSVoucherSerialScope = N'Branch'`.
- Warm or seed branch counters off-peak to avoid first-use fallback scans.

## DevSerial Findings

`DEV_Serial` should not serialize POS save:

- VB6 generated it loosely from daily distinct voucher count.
- Existing data has NULLs, duplicates, gaps, and multiple serials for the same voucher.
- There is no unique index or constraint enforcing it.
- It is display/search legacy data, not the durable accounting identity.

The durable accounting identity remains:

- `Double_Entry_Vouchers_ID`
- `DEV_ID_Line_No`
- `Notes_ID`
- `Transaction_ID`
- debit/credit amounts and account code

The 2026-05-14 decision to remove `POS_DEVSerialAllocator` from the hot path is consistent with the evidence.

## Unnecessary Lock Candidates

Candidate, not automatic change:

- `DEV_Serial` strict allocation: already documented as removable from hot path.
- Company-wide POS voucher serial scope when business meaning is branch/month/type.
- Reference/settings/account reads inside the write transaction.
- Historical `DOUBLE_ENTREY_VOUCHERS` lookup for customer/wallet account inference inside the transaction.
- Cursor-based issue voucher loop inside the main invoice transaction.
- Adjacent `MAX+1 WITH(UPDLOCK,HOLDLOCK)` paths for purchase, stock transfer, manual journals, and POS payment vouchers.

## Recommended Minimal Fixes

Phase 1 safe production mitigations:

1. Deploy the idempotency/duplicate-submit protections already prepared.
2. Deploy `POS_SaveAllocationStageLog` and use it during peak incidents.
3. Remove `DEV_Serial` allocator from the hot path using the reviewed script, preserving accounting keys.
4. Apply `99_POS_VoucherSerialScope.sql` with default `Company`, then explicitly set Kishny to `Branch` only after preflight confirms `sanad_numbering.StoreCoding = 0` for `Sanad_No IN (7,10)`.
5. Monitor `Invoice voucher coding allocation`, `Issue voucher coding allocation`, `Accounting insert`, and `Double entry voucher ID allocation`.
6. Move only clearly stable lookups out of the transaction after comparing returned values before/after in a test DB.

Phase 1 should not alter debit/credit posting, journal row count, customer-visible invoice number format, or issue voucher semantics.

## Phase 2 Deeper Refactor

Phase 2 candidates:

- Redesign `usp_GetNextSerial_V2` to reduce lock duration around `SerialCounters_V2` while preserving format and scope.
- Normalize `SerialCounters_V2` keys to avoid `ISNULL()` in predicates and improve seek quality.
- Pre-seed/warm branch/month/type counters during controlled rollover.
- Standardize adjacent POS writers away from table-wide `MAX+1 WITH(UPDLOCK,HOLDLOCK)`.
- Split reference-data preparation from write transaction using a validated snapshot/preflight pattern.
- Review issue voucher creation order and whether IDs/serials can be allocated before invoice write locks are held.
- Use real production deadlock XML to tune DEV/report indexes and lock order.

## High-Risk Changes To Avoid

Do not rush these:

- Do not remove accounting posting from invoice save.
- Do not make accounting async without a durable outbox, retry, reconciliation, and business approval.
- Do not remove or weaken `NoteSerial1` visibility or format.
- Do not drop constraints/indexes from `DOUBLE_ENTREY_VOUCHERS` without deadlock XML and DBA review.
- Do not switch to `NOLOCK` for correctness-critical save validations.
- Do not replace all serial allocation with loose VB6-style `MAX+1` just because VB6 deadlocked less.
- Do not change store-scoped numbering unless `sanad_numbering.StoreCoding` and business validation prove it.
- Do not rewrite stock issue voucher accounting in the same change as deadlock mitigation.

## Required Production Evidence

Before deeper changes, capture:

- Real `system_health` deadlock XML.
- `POS_SaveAllocationStageLog` p95/p99 around the incident.
- `MANUAL_98_POS_VoucherCoding_Diagnostics.sql` output.
- `MANUAL_90_POS_DoubleEntryVoucher_Diagnostics.sql` output if DEV appears in XML.
- Blocking sessions during peak, especially report/ledger reads.

The evidence should identify exact owner/waiter statements, indexes, lock modes, and objects. Summary counts alone are not enough.

## Final Recommendation

The safest fix path is:

1. Treat voucher coding / `NoteSerial1` allocation as the primary current hotspot.
2. Preserve visible invoice numbering, but scope it to real Kishny business rules, expected to be branch/month/type for `Sanad_No = 7` and `10`.
3. Remove display-only `DEV_Serial` serialization from POS save.
4. Shorten the SQL transaction by moving stable reads and account/settings lookups out of the write window.
5. Preserve synchronous accounting rows for now.
6. Use production deadlock XML before changing `DOUBLE_ENTREY_VOUCHERS` indexes, constraints, or posting order.
