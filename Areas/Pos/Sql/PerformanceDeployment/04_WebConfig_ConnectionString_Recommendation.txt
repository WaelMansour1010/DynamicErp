POS Web.config connection string recommendation
==============================================

Purpose
-------
The 120-worker mixed load test did not show connection pool errors, but the
current connection string uses the .NET default Max Pool Size of 100. Production
has 120+ concurrent users plus report/dashboard requests, so the POS connection
should have explicit headroom.

Recommended additions
---------------------
Add these settings to the POS database connection string:

    Max Pool Size=200;Connect Timeout=30;Pooling=True;

Example
-------
Keep the real server, database, user, and password for each environment. Only
append the tuning options:

    Data Source=YOUR_SQL_SERVER;
    Initial Catalog=Cash;
    User ID=YOUR_APP_LOGIN;
    Password=YOUR_PASSWORD;
    MultipleActiveResultSets=False;
    TrustServerCertificate=True;
    Max Pool Size=200;
    Connect Timeout=30;
    Pooling=True

Important notes
---------------
1. Do not copy the local developer credentials to production.
2. Monitor SQL sessions after deployment. A larger pool is not a license for
   slow queries; it only prevents artificial pool starvation during bursts.
3. If SQL Server is under memory pressure, fix memory/query pressure first
   rather than increasing Max Pool Size further.
4. Keep MultipleActiveResultSets=False unless a specific code path requires it.
